# StudyQuest 🎓

> An AI-powered study planner that tells students **what to study, when to study it** — and makes the process feel like a game.

Built at **Google Developer Group 2026** as a demo of what's possible when you wire Gemini directly into a mobile-first study experience.

---

## What is this?

StudyQuest is an Expo (React Native) app that replaces the anxiety of "I have an exam in two weeks and I don't know where to start" with a structured, adaptive, gamified study plan — generated by Gemini in seconds.

The student provides a subject, their source materials, and a deadline. The AI does everything else: it reads the sources, assesses existing knowledge, and builds a personalised level-by-level roadmap. Progress is tracked through quizzes; if you fail a level, the schedule adjusts. Space repetition keeps earlier topics fresh.

It is **not** a chatbot or AI tutor. The student is in control. The AI is the coach in the background.

## Setup

### Prerequisites

- Node.js 18+
- `npm` or `yarn`
- [Expo Go](https://expo.dev/go) on your phone, or a simulator/emulator

### Clone & install

```bash
git clone https://github.com/viganogabriele/gdg2026.git
cd gdg2026
npm install
```

### Environment variables

Copy the example file and fill in your keys:

```bash
cp .env.example .env.local
```

| Variable | Where to get it | Required |
|---|---|---|
| `EXPO_PUBLIC_GEMINI_API_KEY` | [Google Cloud Console](https://console.cloud.google.com/) → APIs → Generative Language API | ✅ Primary |
| `EXPO_PUBLIC_OPEN_ROUTER_API_KEY` | [openrouter.ai/keys](https://openrouter.ai/keys) | ⬜ Fallback |

> **Note:** Variables must be prefixed with `EXPO_PUBLIC_` to be accessible in the client bundle. If both keys are missing the app falls back to local mock data automatically — the demo always works.

### Run

```bash
# Interactive (scan QR with Expo Go)
npx expo start

# Web browser
npx expo start --web

# iOS simulator
npx expo start --ios

# Android emulator
npx expo start --android
```

## Architecture

```
app/                       # Expo Router file-based routes
  (onboarding)/            # 6-step modal flow: subject → sources → deadline → processing → assessment → roadmap reveal
  (tabs)/                  # Main tabs: Home (today), Roadmap, Profile
  quiz/[id].tsx            # Full-screen level quiz
  focus.tsx                # Landscape focus mode (topic + timer)
  spaced-review.tsx        # Daily spaced repetition session

services/
  gemini.ts                # AI provider layer: Gemini SDK (primary) + OpenRouter REST (fallback)
  api.ts                   # 3-tier fallback: Gemini → OpenRouter → Mock data
  mockData.ts              # Deterministic mock generators (always available)
  spacedRepetition.ts      # Leitner box engine (5 boxes, local, no AI)
  notifications.ts         # Expo notifications scheduling

hooks/
  useStudyStore.ts         # Zustand store + AsyncStorage persistence (all app state)
  useSpacedRepetition.ts   # SR session management
  useGamePoints.ts         # Points / streak / badge triggers

constants/
  theme.ts                 # Dark-mode design system (colors, spacing, typography)
  gamification.ts          # Point values, badge definitions, level thresholds

types/index.ts             # All shared TypeScript types
```

### AI call chain

```
Screen calls api.someFunction()
  │
  ├─ Tier 1: Gemini SDK (EXPO_PUBLIC_GEMINI_API_KEY)
  │    └─ structured output via responseSchema + responseMimeType: application/json
  │
  ├─ Tier 2: OpenRouter REST (EXPO_PUBLIC_OPEN_ROUTER_API_KEY)
  │    └─ json_object response_format (OpenAI-compatible endpoint)
  │
  └─ Tier 3: Local mock data (always available, no keys needed)
```

Each tier logs `✅`, `⚠️`, or `📦` to the console so you can see which provider is serving each request.

## Cognitive Design Choices

### 1. Student control, AI as coach
The design doc is explicit: *"It is not a chatbot or an AI tutor. It is an app that needs to make the student feel 'in control'."* Every AI output (roadmap, quiz, objectives) is presented as the student's plan to execute — not a conversation to have. The AI disappears once onboarding is done.

### 2. Source-centric knowledge ranking
Topics are derived from the student's actual materials, not a generic syllabus. This anchors quiz questions and roadmap levels to real page references — so when the app says "study pages 45–80," it means something the student already has in front of them.

### 3. Assessment-first personalisation
Before generating the roadmap, the app runs a 12-question knowledge assessment. A student scoring 80% on "Calculus" gets the first 2-3 levels pre-marked complete and starts from Derivatives — not Limits. This avoids the classic trap of one-size-fits-all curricula.

### 4. Levels as psychological anchors
The roadmap is split into discrete, completable levels rather than a continuous timeline. Each level has a deadline and a quiz gate. This applies the [goal-gradient effect](https://en.wikipedia.org/wiki/Goal-gradient_hypothesis): the closer you are to completing a level, the more motivated you become. Failing a level pushes future deadlines out — failure is recoverable, not catastrophic.

### 5. Space repetition as a habit, not a feature
The daily spaced-review session (Leitner box 1–5) appears as a mandatory first item in the home screen objectives, not buried in a settings menu. Short (3–5 questions, ~5 minutes) by design — it should feel like a warm-up, not a chore.

### 6. Gamification that respects the student
Points, streaks, and badges exist but are secondary to the roadmap. The leaderboard concept from the design doc was intentionally de-emphasised — competitive pressure during exam prep is harmful for most students. Streaks and badges reward consistency, not performance.

### 7. Focus mode as environment design
Rotating the phone into landscape triggers a minimal focus view: current topic, source reference, and a Pomodoro timer. No navigation, no points, no distractions. The environment itself enforces the study session.

## Tech Stack

| Layer | Choice | Why |
|---|---|---|
| Framework | Expo 54 / React Native 0.81 | iOS + Android + Web from one codebase |
| Routing | Expo Router (file-based) | Clean modal flows, typed routes |
| State | Zustand + AsyncStorage | Lightweight, persists across sessions, no boilerplate |
| AI primary | Google Gemini 2.0 Flash (via `@google/generative-ai` SDK) | Fast, cheap, excellent structured output |
| AI fallback | OpenRouter → `google/gemini-2.0-flash-001` | Same model, different auth — maximises uptime |
| Structured output | `responseSchema` + `responseMimeType: application/json` | No regex parsing; Gemini returns type-safe JSON |
| Animations | React Native Reanimated 4 | 60fps animations on the JS thread |
| Notifications | Expo Notifications | Cross-platform local push |

## What We'd Do With Another Week

### 1. Real source ingestion
Right now "add sources" accepts PDF files and URLs but doesn't extract their text — the AI generates content purely from the subject title. With another week: run PDFs through Gemini's multimodal `inlineData` to extract actual content, and scrape URLs via a lightweight server function. This would make quiz questions reference real passages.

### 2. Streaming responses
The processing screen shows a spinner for 3–10 seconds. Gemini's `streamGenerateContent` endpoint would let us stream topic sections as they're generated, showing the roadmap building in real-time — far more engaging for a demo.

### 3. Multi-subject support
The store and type system already support multiple subjects (`subjects: Subject[]`) but the UI only creates one. Adding a subject switcher + home screen that aggregates objectives across subjects is 2–3 days of UI work.

### 4. Adaptive rescheduling via Gemini
`adjustRoadmap` is currently pure local logic (push deadlines by ±2 days). With another week: send the full context to Gemini — study history, quiz scores, time available, deadline — and get a genuinely intelligent reschedule that accounts for topic dependencies.

### 5. Push notifications
The `notifications.ts` service is implemented but never scheduled. Wiring up daily reminders, deadline warnings, and streak-break alerts is a half-day integration task — the infrastructure is already there.

### 6. Backend + auth
Every API key lives in the client bundle. For production: a lightweight backend (Cloudflare Workers or Next.js API routes) would proxy Gemini calls, add per-user rate limiting, and store progress server-side so data survives app reinstalls.
